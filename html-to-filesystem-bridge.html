HTML → Filesystem Bridge

This document describes how an HTML document rendered inside LiveHTMLPreview can persist arbitrary files back into its workspace project. The HTML cannot touch the host filesystem directly — all writes are forwarded through the React parent frame to the authenticated content-management API on the backend.

Architecture

┌─────────────────────┐     postMessage     ┌────────────────────┐     PUT (JWT)     ┌──────────────────────┐
│  iframe HTML / JS   │ ──────────────────▶ │ LiveHTMLPreview.jsx│ ────────────────▶ │ /api/workspace/:p/   │
│  (sandboxed)        │ ◀────────────────── │  (parent window)   │ ◀──────────────── │   files/bridge-write │
└─────────────────────┘   write:result      └────────────────────┘   { path, bytes } └──────────────────────┘
                                                                                                │
                                                                                                ▼
                                                                                     workspace/<project>/...

The iframe never sees the JWT. The parent already holds it and signs the request via the existing apiAxios client.

Quick start (from inside an iframe HTML)

function bridgeWrite(path, content, encoding) {
  return new Promise((resolve, reject) => {
    if (window.parent === window) return reject(new Error('No parent frame'));
    const requestId = 'w_' + Date.now() + '_' + Math.random().toString(36).slice(2, 8);
    const onMessage = (event) => {
      const m = event.data;
      if (!m || m.type !== 'workspace:write:result' || m.requestId !== requestId) return;
      window.removeEventListener('message', onMessage);
      m.ok ? resolve(m.result) : reject(new Error(m.error));
    };
    window.addEventListener('message', onMessage);
    window.parent.postMessage({
      type: 'workspace:write',
      requestId,
      path,
      content,
      encoding: encoding || 'utf-8',
    }, '*');
  });
}

// Examples
await bridgeWrite('./selection.json', JSON.stringify({ pick: 'Mar' }, null, 2));
await bridgeWrite('/state/last-run.json', JSON.stringify({ ts: Date.now() }));
await bridgeWrite('./screenshot.png', base64String, 'base64');

Path resolution

Path formResolves toExample (HTML at html-json/dashboard.html)
/foo/bar.jsonProject root→ foo/bar.json
./bar.jsonHTML file's folder→ html-json/bar.json
bar.jsonHTML file's folder (same as above)→ html-json/bar.json
./sub/bar.jsonBelow HTML's folder→ html-json/sub/bar.json

Parent directories are created automatically. Existing files are always overwritten.

Path traversal is blocked. Any path that resolves above the project root (e.g. ../../etc/hosts) is rejected by both the frontend resolver and a second check in the backend service.

Encoding

  • UTF-8 (default) — pass any string. The backend writes it as a UTF-8 buffer.
  • base64 — set encoding: 'base64' for binary content (images, PDFs, zips). Encode the bytes yourself, e.g. via btoa() or FileReader.readAsDataURL() (then strip the data:...;base64, prefix).

Message protocol

Request: iframe → parent

{
  type: 'workspace:write',
  requestId: 'w_1715000000_abc123',  // any unique string
  path: './selection.json',           // see "Path resolution"
  content: '...',                     // string (utf-8 text or base64)
  encoding: 'utf-8' | 'base64'        // default 'utf-8'
}

Response: parent → iframe

// success
{
  type: 'workspace:write:result',
  requestId: 'w_1715000000_abc123',
  ok: true,
  result: { success: true, message: 'File written: html-json/selection.json',
            path: 'html-json/selection.json', bytes: 73 }
}

// failure
{
  type: 'workspace:write:result',
  requestId: 'w_1715000000_abc123',
  ok: false,
  error: 'Path escapes project root: ../../etc/hosts'
}

Security model

  • Only messages whose event.source matches the previewer's iframe are processed. Messages from other windows or other iframes on the same page are ignored.
  • The JWT never leaves the parent frame.
  • Path traversal is blocked in two places: the frontend resolver collapses .. and refuses paths that climb above the project; the backend resolves the absolute path and requires it to start with the project root.
  • Writes are unconditional overwrites. There is no delete, no append, no rename via this bridge — those remain on the existing endpoints.
  • Any file extension is allowed. The backend does not interpret content; it writes the bytes.
Beware of the overwrite semantics. An HTML page that writes to a fixed path on every load can clobber user data. Use unique filenames or read-modify-write patterns (read via fetch('./state.json'), mutate, then write back) when you need accumulation.

Working example

The dashboard at workspace/web-test/html-json/dashboard.html uses the bridge: clicking any monthly bar writes html-json/selection.json with the chosen month and revenue, and shows the result inline.

Limitations

  • One-way. The bridge writes only. To read, use fetch('./file.json') against the same content API the preview already serves.
  • Synchronous-feeling but async. Each write is a real HTTP call, so latency depends on the backend. The 10-second timeout in the helper is configurable.
  • No partial writes. Files are written in one shot; there is no streaming.
  • Single-iframe scope. Only the previewer's iframe can use this. Nested iframes inside the preview are not granted access.
